## 1. 개요 (Executive Summary)

### 1.1 문제 정의 (Problem Statement)
공공기관의 민원 데이터는 민감한 개인정보를 포함하고 있어 외부 클라우드 API(ChatGPT, Claude 등)를 사용할 수 없으며, 기존의 단순 키워드 기반 분류 방식으로는 복잡한 민원 맥락을 이해하고 적절한 답변을 생성하기 어렵습니다. 담당 공무원은 수많은 민원을 수작업으로 처리하며 유사 사례를 검색하는 데 막대한 시간을 소비하고 있습니다.

### 1.2 제안 솔루션 (Proposed Solution)
폐쇄망(온프레미스) 환경에서 구동 가능한 경량화된 LLM(sLLM)을 활용하여 민원을 자동으로 분류하고, 과거 답변 데이터를 학습해 고품질의 표준 답변 초안을 즉시 생성하는 On-Device AI 시스템을 개발합니다. 공개 민원 데이터를 크롤링하여 도메인 특화 파인튜닝을 실시하고, FAISS 기반 벡터 검색으로 유사 사례를 빠르게 참조할 수 있는 통합 솔루션을 제공합니다.

### 1.3 핵심 비즈니스 영향 (Key Business Impact)
- **업무 효율성 향상**: 민원 처리 시간을 평균 60% 이상 단축 (수작업 대비)
- **보안 및 개인정보 보호**: 데이터 외부 유출 없이 100% 폐쇄망 내 처리
- **비용 절감**: 대규모 클라우드 API 비용 없이 저비용 온프레미스 운영 (연간 유지비 90% 절감 예상)
- **일관성 있는 답변**: 표준 서식 기반 답변 생성으로 품질 편차 최소화
- **확장성**: 타 지자체 및 공공기관으로 즉시 이식 가능한 Docker 컨테이너 기반 배포

## 2. 배경 및 컨텍스트 (Background & Context)

### 2.1 현재 상태 및 문제점 (Current State & Pain Points)

#### 문제점 1: 개인정보 보호 및 보안 규제
- 공공기관 민원 데이터는 주민등록번호, 주소, 연락처 등 민감 정보 포함
- 클라우드 기반 AI 서비스(ChatGPT, Claude API) 사용 시 개인정보보호법 위반 리스크
- 폐쇄망 환경에서만 처리 가능하도록 제한됨

#### 문제점 2: 기존 키워드 기반 분류의 한계
- 단순 키워드 매칭은 민원의 복잡한 맥락과 의도를 파악하지 못함
- 오분류율 높음 (예: "도로 파손"과 "보도 파손"을 구분 못함)
- 동음이의어, 신조어, 지역 방언 등에 대응 불가

#### 문제점 3: 비효율적인 수작업 프로세스
- 담당자가 민원을 읽고 분류하며 과거 유사 사례를 수작업으로 검색
- 민원 1건당 평균 15-30분 소요 (답변 작성 포함)
- 담당자의 경험과 숙련도에 따라 답변 품질 편차 큼

#### 문제점 4: 유사 사례 검색의 어려움
- 기존 키워드 검색으로는 의미적으로 유사한 사례를 찾기 어려움
- 과거 답변 데이터베이스가 체계적으로 활용되지 못함

### 2.2 시장 및 사용자 조사 인사이트 (Market & User Research Insights)

#### 대상 사용자
- **1차 사용자**: 지자체 민원 담당 공무원 (행정직 5-7급)
- **2차 수혜자**: 민원을 제출하는 시민 (빠른 답변 수령)
- **기술 관리자**: 공공기관 IT 관리자 (시스템 설치 및 유지보수)

#### 시장 환경
- 국내 지자체 약 243개 (광역 17개, 기초 226개)
- 연간 민원 처리 건수: 약 1,500만 건 이상 (행정안전부 민원포털 기준)
- 공공기관 AI 도입 정책: 행안부 '공공 AI 윤리 가이드라인' 준수 필요

### 2.3 전략적 정렬 (Strategic Alignment)
- **동아대학교 SW중심대학사업**: 현장미러형 연계 프로젝트 (실증적 AI/SW 프로젝트)
- **정부 디지털 전환 정책**: 공공 서비스의 AI 활용 촉진
- **개인정보보호 강화**: 데이터 주권 및 로컬 AI 처리 트렌드 부합

## 3. 목표 및 성공 지표 (Goals & Success Metrics)

### 3.1 주요 목표 (Primary Objectives)

#### OBJ-001: 민원 처리 시간 단축
- **측정 가능한 목표**: 민원 답변 초안 생성 시간을 평균 15분에서 3분 이하로 단축 (80% 감소)

#### OBJ-002: 민원 자동 분류 정확도 향상
- **측정 가능한 목표**: 민원 카테고리 분류 정확도 85% 이상 달성 (기존 키워드 방식 대비 30%p 향상)

#### OBJ-003: 폐쇄망 환경에서 안정적 운영
- **측정 가능한 목표**: 인터넷 연결 없이 100% 온프레미스 환경에서 정상 동작

#### OBJ-004: 유사 사례 검색 시간 단축
- **측정 가능한 목표**: 관련 과거 사례 검색 시간을 평균 5분에서 10초 이하로 단축 (97% 감소)

### 3.2 핵심 성과 지표 (Key Performance Indicators)

#### KPI-001: 답변 생성 속도
- **지표**: 민원 입력 후 답변 초안 생성까지 소요 시간
- **목표값**: p50 < 2초, p95 < 5초 (폐쇄망 서버 환경: CPU 16코어, RAM 32GB 기준)

#### KPI-002: 민원 분류 정확도 (Accuracy)
- **지표**: 시스템이 제안한 카테고리와 담당자의 최종 선택이 일치하는 비율
- **목표값**: ≥ 85% (테스트 데이터셋 1,000건 기준)

#### KPI-003: 답변 품질 만족도
- **지표**: 담당자가 생성된 답변 초안을 "그대로 또는 최소 수정 후 사용" 비율
- **목표값**: ≥ 70% (사용자 피드백 조사)

#### KPI-004: 시스템 응답 가용성
- **지표**: 시스템 정상 가동 시간 비율 (Uptime)
- **목표값**: ≥ 99.5% (주간 업무시간 기준)

#### KPI-005: 벡터 검색 정확도 (Recall@5)
- **지표**: 상위 5개 유사 사례 중 실제 관련 사례가 포함된 비율
- **목표값**: ≥ 80%

#### KPI-006: 모델 추론 메모리 사용량
- **지표**: LLM 추론 시 GPU/CPU 메모리 사용량
- **목표값**: ≤ 8GB VRAM (GPU 사용 시), ≤ 16GB RAM (CPU 전용 시)

### 3.3 성공 기준 (Success Criteria)
- **Phase 1 (MVP)**: KPI-001, KPI-002 목표 달성 + 핵심 기능 구현 완료
- **Phase 2 (최종)**: 모든 KPI 목표 달성 + 사용자 수용 테스트(UAT) 통과

## 4. 사용자 스토리 및 요구사항 (User Stories & Requirements)

### 4.1 사용자 페르소나 (User Personas)

#### 페르소나 1: 김민원 (민원 담당 공무원, 35세, 행정직 6급)
- **특징**: 하루 평균 20-30건의 민원 처리, IT 활용 중급 수준
- **목표**: 민원 처리 시간 단축, 정확한 답변 작성, 민원인 만족도 향상
- **불편사항**: 유사 사례 검색 어려움, 답변 작성 피로도, 표준 서식 숙지 부담

#### 페르소나 2: 박시스템 (IT 관리자, 42세, 전산직 5급)
- **특징**: 지자체 내부 시스템 관리, 보안 및 개인정보 보호 책임
- **목표**: 안정적인 시스템 운영, 보안 규정 준수, 유지보수 최소화
- **불편사항**: 복잡한 설치 과정, 외부 의존성 관리, 문서 부족

### 4.2 사용자 스토리 (User Stories)

#### US-001: 민원 자동 분류
**As a** 민원 담당 공무원
**I want** 민원 텍스트를 입력하면 자동으로 적절한 카테고리로 분류되는 기능
**So that** 수작업으로 분류 기준을 고민하는 시간을 절약할 수 있다

#### US-002: 답변 초안 자동 생성
**As a** 민원 담당 공무원
**I want** 민원 내용을 입력하면 표준 서식에 맞춰 답변 초안이 즉시 생성되는 기능
**So that** 답변 작성 시간을 단축하고 일관된 품질의 답변을 제공할 수 있다

#### US-003: 유사 사례 검색
**As a** 민원 담당 공무원
**I want** 현재 민원과 의미적으로 유사한 과거 사례를 자동으로 추천받는 기능
**So that** 이전 답변을 참고하여 더 정확하고 신속하게 대응할 수 있다

#### US-004: 답변 수정 및 복사
**As a** 민원 담당 공무원
**I want** 생성된 답변 초안을 직접 수정하고 클립보드에 복사하는 기능
**So that** 생성된 답변을 기반으로 최종 검토 후 즉시 민원 시스템에 붙여넣을 수 있다

#### US-005: 폐쇄망 환경 설치
**As an** IT 관리자
**I want** 인터넷 연결 없이 Docker 컨테이너로 시스템을 설치하고 실행하는 기능
**So that** 보안 규정을 준수하면서 간편하게 배포할 수 있다

#### US-006: 학습 데이터 관리
**As an** IT 관리자
**I want** 추가 민원 데이터를 업로드하여 모델을 재학습시키는 기능
**So that** 우리 지자체의 특성에 맞게 시스템을 지속적으로 개선할 수 있다

### 4.3 기능 요구사항 (Functional Requirements)

#### 4.3.1 데이터 수집 및 전처리

##### FR-001: 공개 민원 데이터 크롤링
- 각 지자체 '전자민원창구' 게시판에서 공개 민원 및 답변 데이터 자동 수집
- 크롤링 대상: 민원 제목, 본문, 답변, 카테고리, 작성일자
- 수집 규모: 최소 10,000건 이상의 민원-답변 쌍

##### FR-002: 데이터 비식별화 처리
- 수집된 데이터에서 개인정보 자동 탐지 및 마스킹
- 제거 대상: 이름, 주민등록번호, 전화번호, 이메일, 주소 등
- 처리 방법: 정규표현식 + NER 모델 활용

##### FR-003: 데이터 품질 검증
- 중복 데이터 제거 (해시 기반 중복 탐지)
- 짧거나 의미 없는 텍스트 필터링 (민원 본문 최소 20자 이상)
- 데이터 무결성 검증 (민원-답변 쌍 매칭 확인)

##### FR-004: 학습 데이터 포맷 변환
- 수집된 데이터를 파인튜닝 형식으로 변환 (Instruction-Response 형태)
- 포맷 예시:
  ```json
  {
    "instruction": "다음 민원에 대해 표준 서식으로 답변을 작성하세요.",
    "input": "[민원 본문]",
    "output": "[표준 답변]"
  }
  ```

#### 4.3.2 모델 학습 및 최적화

##### FR-005: 오픈소스 sLLM 선정 및 다운로드
- 후보 모델: Gemma 2 (2B/7B), Llama 3 (8B), Qwen 2.5 (7B), Phi-3 등
- 선정 기준: 한국어 성능, 모델 크기, 라이선스, 추론 속도
- HuggingFace Hub에서 모델 가중치 다운로드

##### FR-006: Supervised Fine-Tuning (SFT) 실행
- LoRA 또는 QLoRA 기법을 사용한 효율적 파인튜닝
- 학습 데이터: 크롤링된 민원-답변 쌍 (80% train, 10% validation, 10% test)
- 하이퍼파라미터 튜닝: Learning rate, Batch size, Epoch, LoRA rank 등
- 학습 환경: GPU 지원 (CUDA) 또는 CPU fallback

##### FR-007: 모델 Quantization (양자화)
- INT8 또는 INT4 양자화로 모델 크기 및 추론 속도 최적화
- 도구: ONNX Runtime, TensorRT, llama.cpp 등
- 목표: 모델 크기 50% 이상 감소, 추론 속도 2배 이상 향상

##### FR-008: 모델 평가 및 벤치마킹
- 평가 지표: Accuracy, F1-Score, BLEU/ROUGE (답변 품질), Perplexity
- 테스트 데이터셋 (1,000건) 기준 성능 측정
- 기존 키워드 방식 대비 정량적 비교

#### 4.3.3 답변 생성 엔진

##### FR-009: 표준 답변 서식 시스템 프롬프트 설계
- 공공기관 답변 표준 서식 정의 (인사말, 본문, 담당부서, 연락처 등)
- 프롬프트 엔지니어링: Few-shot examples, Chain-of-Thought 등 적용
- 톤앤매너 일관성 유지 (공식적, 공손한 표현)

##### FR-010: 민원 분류 기능
- 입력: 민원 본문 텍스트
- 출력: 분류 카테고리 (예: 도로/교통, 환경/위생, 복지, 세금, 기타)
- 방법: LLM 기반 Zero-shot 또는 Few-shot 분류

##### FR-011: 답변 초안 생성 기능
- 입력: 민원 본문 + 분류 카테고리 (선택적)
- 출력: 표준 서식 기반 답변 초안 (200-500자)
- 생성 파라미터: Temperature, Top-p, Max tokens 조정 가능

##### FR-012: 답변 생성 시 유사 사례 참조 (RAG)
- FAISS 벡터 검색으로 상위 3-5개 유사 사례 검색
- 검색된 사례를 컨텍스트로 LLM에 제공
- RAG 파이프라인: Retrieval → Augmentation → Generation

#### 4.3.4 유사 사례 검색 (벡터 검색)

##### FR-013: 문서 임베딩 생성
- 모든 과거 민원-답변 데이터를 벡터로 변환
- 임베딩 모델: KoSimCSE, KoBERT, Sentence-Transformers (다국어 모델) 등
- 벡터 차원: 384-768 차원

##### FR-014: FAISS 인덱스 구축
- FAISS 라이브러리를 사용한 벡터 인덱스 생성
- 인덱스 타입: IndexFlatL2 (소규모) 또는 IndexIVFFlat (대규모)
- 인덱스 저장 및 로드 기능

##### FR-015: 유사도 검색 API
- 입력: 민원 본문 텍스트
- 출력: 상위 K개의 유사 민원-답변 쌍 (K=5 기본값)
- 유사도 점수 함께 반환 (Cosine similarity 또는 L2 distance)

##### FR-016: 검색 결과 필터링
- 유사도 임계값 설정 (Threshold) → 너무 낮은 유사도는 제외
- 중복 결과 제거 (동일 민원이 여러 번 검색되지 않도록)

#### 4.3.5 온프레미스 서빙 최적화

- **대체 방안**: CPU 전용 모드 지원 (속도는 느림)

### 6.4 통합 요구사항 (Integration Requirements)

#### INT-001: 기존 시스템과의 독립성
- 현재 프로젝트는 **독립형(standalone)** 시스템
- 기존 민원 시스템과 직접 통합 없음 (범위 외)
- 향후 REST API를 통한 연동 가능하도록 설계

#### INT-002: 데이터 입출력 인터페이스
- **입력**: 텍스트 (민원 본문)
- **출력**: JSON 형태 (답변 초안, 유사 사례)
- **파일 업로드**: CSV/Excel 파일 업로드 지원 (배치 처리, 선택적)

### 6.5 데이터 요구사항 (Data Requirements)

#### DATA-001: 학습 데이터
- **소스**: 각 지자체 공개 민원 게시판
- **수량**: 최소 10,000건 (목표 20,000건)
- **형식**: JSON 또는 CSV
- **필드**: 민원 ID, 제목, 본문, 카테고리, 답변, 작성일자

#### DATA-002: 벡터 인덱스 데이터
- **소스**: 학습 데이터와 동일
- **형식**: FAISS 인덱스 파일 (.index) + 메타데이터 (JSON)
- **크기**: 약 1-5GB (10만 건 기준)

#### DATA-003: 설정 데이터
- **형식**: YAML 또는 JSON
- **내용**: 모델 경로, API 포트, 프롬프트 템플릿, 카테고리 목록 등

### 6.6 기술적 제약사항 (Technical Constraints)

#### CONS-001: 폐쇄망 환경
- 인터넷 연결 불가 → 모든 의존성을 사전 패키징
- 외부 API 호출 금지 (OpenAI, Claude 등)

#### CONS-002: 하드웨어 제약
- GPU 없는 환경도 지원 필요 (CPU 전용 모드)
- 메모리 제한: 16GB RAM 이하 환경에서도 작동

#### CONS-003: 보안 규정
- 개인정보보호법 준수 (데이터 비식별화)
- 접근 로그 기록 (감사 추적)

#### CONS-004: 모델 라이선스
- 오픈소스 라이선스 준수 (Apache 2.0, MIT 등)
- 상용 사용 가능 모델만 선정 (Gemma 2, Llama 3 등)

## 8. 수락 기준 (Acceptance Criteria)

### 8.1 기능별 수락 기준 (Feature-Specific Acceptance Criteria)

#### AC-001: 데이터 크롤링 및 전처리
**Given** 지자체 민원 게시판 URL이 주어졌을 때
**When** 크롤러를 실행하면
**Then**
- 최소 10,000건 이상의 민원-답변 쌍이 수집된다
- 수집된 데이터 중 개인정보(이름, 주민번호, 전화번호 등)가 마스킹 처리된다
- 중복 데이터가 제거되고 유효한 데이터만 남는다
- 최종 데이터가 JSON 또는 CSV 형식으로 저장된다

#### AC-002: 모델 파인튜닝
**Given** 전처리된 학습 데이터가 준비되었을 때
**When** SFT 파인튜닝을 실행하면
**Then**
- 선택한 sLLM (Gemma 2 또는 Llama 3)이 성공적으로 로드된다
- LoRA/QLoRA 파인튜닝이 에러 없이 완료된다
- 테스트 데이터셋(1,000건)에서 답변 생성 정확도가 BLEU > 0.3 또는 ROUGE-L > 0.4를 달성한다
- 파인튜닝된 모델이 로컬에 저장된다

#### AC-003: 답변 초안 생성
**Given** 민원 본문이 입력되었을 때
**When** "답변 생성" 버튼을 클릭하면
**Then**
- 2-5초 이내에 답변 초안이 생성된다
- 생성된 답변은 표준 서식(인사말, 본문, 담당부서, 연락처)을 포함한다
- 답변 길이는 200-500자 사이다
- 생성된 답변이 화면에 표시된다

#### AC-004: 민원 자동 분류
**Given** 민원 본문이 입력되었을 때
**When** 시스템이 민원을 분석하면
**Then**
- 적절한 카테고리(도로/교통, 환경/위생, 복지, 세금, 기타 중 하나)가 자동으로 제안된다
- 분류 정확도가 85% 이상이다 (테스트 데이터 1,000건 기준)

#### AC-005: 유사 사례 검색
**Given** 민원 본문이 입력되었을 때
**When** 벡터 검색을 실행하면
**Then**
- 상위 5개의 유사 민원-답변 쌍이 검색된다
- 검색 시간이 1초 이내다
- 각 결과에 유사도 점수가 표시된다
- Recall@5가 80% 이상이다 (관련 사례가 상위 5개 안에 포함되는 비율)

#### AC-006: 클립보드 복사
**Given** 답변 초안이 생성되었을 때
**When** "클립보드에 복사" 버튼을 클릭하면
**Then**
- 답변 텍스트가 클립보드에 복사된다
- "복사되었습니다" 확인 메시지가 표시된다
- Ctrl+V로 다른 곳에 붙여넣기 가능하다

#### AC-007: Docker 배포
**Given** Docker가 설치된 폐쇄망 서버가 준비되었을 때
**When** `docker-compose up -d` 명령을 실행하면
**Then**
- 모든 컨테이너가 에러 없이 시작된다
- 웹 UI가 지정된 포트(예: 8501)에서 접근 가능하다
- 인터넷 연결 없이도 정상 작동한다
- 모델 및 벡터 인덱스가 정상적으로 로드된다

## 10. 리스크 및 완화 전략 (Risks & Mitigations)

### 10.1 기술적 리스크 (Technical Risks)

#### RISK-001: 모델 성능 미달
**설명**: 파인튜닝된 sLLM이 기대한 답변 품질을 생성하지 못할 위험

**영향**: 높음 (프로젝트 핵심 목표)

**발생 가능성**: 중간

**완화 전략**:
- **사전 예방**: 벤치마크 데이터셋으로 사전 테스트 (Gemma 2, Llama 3 비교)
- **대응 방안 1**: 더 많은 학습 데이터 수집 (20,000건 이상)
- **대응 방안 2**: 프롬프트 엔지니어링 개선 (Few-shot examples, CoT)
- **대응 방안 3**: 더 큰 모델로 변경 (7B → 13B, 단 추론 속도 감소)
- **최종 수단**: RAG 기반 템플릿 매칭으로 폴백 (검색된 유사 사례를 수정하여 제공)

---

#### RISK-002: 추론 속도 느림
**설명**: 폐쇄망 서버의 하드웨어 제약으로 답변 생성 시간이 10초 이상 소요

**영향**: 중간 (사용자 경험 저하)

**발생 가능성**: 높음

**완화 전략**:
- **사전 예방**: Quantization (INT4/INT8) 적극 활용
- **대응 방안 1**: 모델 크기 축소 (7B → 2B)
- **대응 방안 2**: Ollama의 캐싱 기능 활용
- **대응 방안 3**: GPU 서버 요청 (멘토 교수 또는 학교 지원)
- **최종 수단**: "빠른 모드"와 "정확한 모드" 분리 (사용자 선택)

---

#### RISK-003: 벡터 검색 정확도 낮음
**설명**: FAISS 검색 결과가 실제로 유사하지 않은 사례를 반환

**영향**: 중간 (RAG 품질 저하)

**발생 가능성**: 중간

**완화 전략**:
- **사전 예방**: 고품질 임베딩 모델 선정 (KoSimCSE, multilingual-e5)
- **대응 방안 1**: 하이브리드 검색 (벡터 + 키워드 BM25)
- **대응 방안 2**: 유사도 임계값 조정 (낮은 점수 결과 필터링)
- **대응 방안 3**: 사용자 피드백 기반 재학습 (클릭 로그 활용)

---

#### RISK-004: 개인정보 유출
**설명**: 비식별화 처리 실패로 개인정보가 모델 출력에 노출

**영향**: 매우 높음 (법적 문제, 프로젝트 중단)

**발생 가능성**: 낮음

**완화 전략**:
- **사전 예방**: 다층 비식별화 (정규표현식 + NER 모델 + 수동 검증)
- **대응 방안 1**: 출력 텍스트 후처리 필터링 (민감 정보 재탐지)
- **대응 방안 2**: 학습 데이터 재검증 (팀원 교차 확인)
- **대응 방안 3**: 로그 및 출력 모니터링 (정기적 감사)
- **최종 수단**: 공개 데이터만 사용 (실제 개인정보 제외)

## 11. 부록 (Appendix)

### 11.1 용어집 (Glossary)

| 용어 | 정의 |
|------|------|
| **sLLM** | Small Language Model - 파라미터 수가 10B 미만인 경량화된 언어 모델 |
| **SFT** | Supervised Fine-Tuning - 라벨된 데이터를 사용한 지도 학습 기반 파인튜닝 |
| **LoRA** | Low-Rank Adaptation - 효율적인 파인튜닝 기법 (전체 모델 대신 일부 레이어만 학습) |
| **QLoRA** | Quantized LoRA - LoRA에 양자화를 결합하여 메모리 사용량을 더욱 줄인 기법 |
| **FAISS** | Facebook AI Similarity Search - 대규모 벡터 유사도 검색 라이브러리 |
| **RAG** | Retrieval-Augmented Generation - 검색 결과를 활용하여 답변 생성 품질을 향상시키는 기법 |
| **Quantization** | 양자화 - 모델 가중치를 낮은 정밀도(INT8, INT4)로 변환하여 크기 및 속도 최적화 |
| **Embedding** | 임베딩 - 텍스트를 고차원 벡터로 변환한 수치 표현 (의미적 유사도 계산 가능) |
| **Ollama** | 로컬 환경에서 LLM을 쉽게 실행할 수 있는 오픈소스 도구 |
| **Streamlit** | Python 기반 웹 UI 프레임워크 (데이터 사이언스 및 ML 앱 구축에 특화) |
| **폐쇄망** | 인터넷 연결이 차단된 내부 네트워크 환경 (온프레미스) |
| **비식별화** | 개인정보를 식별할 수 없도록 가명 처리 또는 마스킹하는 작업 |
